# OSM + GTFS Routing Service

> **Статус в PWA:** планировщик маршрутов A→B в React-версии в разработке.  
> Текущий поиск — по GTFS-графу в legacy `app.html`. Этот документ описывает **будущий** backend-сервис.

## Задача

Нужен отдельный сервис, который:

- загружает дорожный граф из OSM
- читает GTFS-фид
- привязывает остановки GTFS к дорожному графу
- добавляет остановки как новые узлы графа
- связывает GTFS-маршруты с дорогами
- отдает подготовленный граф и диагностические данные наружу

Это действительно похоже на стек вокруг OSRM, но не совпадает с ним один в один.

## Почему одного OSRM недостаточно

OSRM хорошо решает задачу "кратчайший путь по дорожному графу", но сам по себе не закрывает несколько важных требований:

- GTFS-остановки не становятся его доменными сущностями автоматически
- граф нельзя удобно модифицировать на лету под ваши stop-узлы
- GTFS-shapes не являются гарантированно валидной дорожной геометрией
- оптимизация маршрутной сети и диагностика отклонений между GTFS и OSM требуют отдельного preprocessing-слоя

Итог: OSRM можно использовать как backend для дорожных маршрутов, но слой `GTFS -> snap -> split -> augment graph` лучше выносить в отдельный сервис.

## Рекомендуемая схема

```text
OSM extract/pbf
    ->
road graph builder
    ->
GTFS ingest
    ->
stop snap + edge split
    ->
shape map matching
    ->
route diagnostics / optimization
    ->
API for PWA / analytics / export
```

## Основные компоненты

### 1. OSM Graph Builder

На входе:

- `.osm.pbf` по Камчатке или нужному bbox

На выходе:

- граф дорог: `nodes + edges + geometry + tags`

Источники для этого слоя:

- собственный импорт из OSM
- GraphHopper/Valhalla preprocessing
- pgRouting/PostGIS

### 2. GTFS Ingest

Из GTFS нужны как минимум:

- `stops.txt`
- `routes.txt`
- `trips.txt`
- `stop_times.txt`
- `shapes.txt`

На этом этапе строятся:

- справочник остановок
- шаблоны маршрутов по последовательности остановок
- сопоставление `route_id -> trips -> shape_id`

### 3. Stop Snap + Edge Split

Ключевая операция:

1. найти ближайшее дорожное ребро для остановки
2. спроецировать остановку на геометрию ребра
3. разрезать ребро в точке проекции
4. создать новый узел остановки
5. создать новые ребра `A -> stop`, `stop -> B`

Именно этот шаг делает остановку частью routable-графа, а не просто точкой на карте.

### 4. Shape Map Matching

После врезки остановок в граф нужно связать GTFS-маршрут с реальными дорогами:

- map matching по `shapes.txt`
- контроль прохождения маршрута через ожидаемые stop-узлы
- поиск разрывов и "телепортов" между shape и дорожной сетью

### 5. Optimization Layer

Слово "оптимизирует" здесь может означать разное. Практически полезно разделить на три режима:

- `topology optimization`: выправить геометрию маршрута по OSM
- `diagnostics optimization`: найти проблемные shape-сегменты, которые не ложатся на дорогу
- `network optimization`: оценить дублирование маршрутов, длинные заезды, лишние петли, плохие пересадки

Первый режим можно запускать сразу. Третий обычно требует отдельной транспортной логики и KPI.

## Что лучше взять как движок

### Вариант A. Отдельный preprocessing-сервис + свой граф

Плюсы:

- полный контроль над stop-узлами
- удобно делать диагностику и экспорт
- подходит для кастомной логики поверх GTFS

Минусы:

- больше собственной логики

### Вариант B. Valhalla

Плюсы:

- лучше приспособлен к multimodal/mapping-задачам, чем чистый OSRM
- удобнее расширять для маршрутизации с пешеходными сегментами

Минусы:

- все равно нужен свой слой подготовки GTFS-остановок

### Вариант C. GraphHopper

Плюсы:

- сильный routing-движок
- удобен, если основной фокус именно на дорожном графе

Минусы:

- stop augmentation и GTFS-диагностика все равно остаются отдельной задачей

### Вариант D. OSRM

Подходит, если:

- нужен очень быстрый road routing
- stop augmentation выполняется заранее вне OSRM

Не подходит как единственный компонент, если вы хотите именно "врезать" GTFS-остановки в граф и анализировать маршрутную сеть.

## Практический вывод

Для вашего кейса оптимальная конструкция такая:

1. отдельный сервис preprocessing
2. загрузка OSM-графа
3. загрузка GTFS
4. добавление stop-узлов через разрезание ребер
5. map matching `shapes.txt` к дорожному графу
6. наружный API для PWA и аналитики

Если позже понадобится production-routing для пользователя, тогда:

- либо подключаем Valhalla
- либо подключаем OSRM как нижний road-routing слой

## Что добавлено в репозиторий

В `services/gtfs_osm_router/` лежит минимальный каркас такого сервиса на Python:

- загрузка GTFS
- построение шаблонов маршрутов
- загрузка графа в JSON
- добавление остановок как новых узлов графа
- HTTP API для локального запуска

Это не заменяет полноценный OSM-импортер, но уже закрывает критичный слой `GTFS -> graph augmentation`.
